# eTutor

eTutor is a secure role-based web application for managing personal tutoring in a university environment. It supports students, tutors, staff, and admin-level staff through a PHP/MySQL backend and a multi-page HTML/CSS/JavaScript frontend.

The system covers tutor allocation and reallocation, student-tutor messaging, meetings, document upload and feedback, blog posts and comments, dashboards, reporting, activity logging, notification emails, and inactivity monitoring.

## Project Overview

This project was built for an enterprise-style university tutoring workflow where:

- every student should have a personal tutor
- authorised staff can allocate, reallocate, and bulk allocate tutors
- students and tutors interact inside the system rather than through direct email
- emails are used only for notifications generated by backend events
- staff and admin users can monitor activity, dashboards, and reports

The application entry point redirects directly to the login page:

```text
http://localhost/eTutor/
```

which forwards to:

```text
/Frontend/Pages/Auth/Login.html
```

## Core Features

### Authentication and Accounts

- secure login with JWT access and refresh tokens
- role-based access control for student, tutor, staff, and admin-level staff
- last login date/time display
- first-login welcome flow
- forced password change support
- temporary login lockout after repeated failed password attempts

### Student Features

- view allocated personal tutor
- personal dashboard summarising interaction with tutor
- send and receive messages with assigned tutor
- upload documents
- view document feedback from tutor
- create, view, and delete blog posts
- comment on blog posts
- view meetings and meeting outcomes
- manage own profile

### Tutor Features

- dashboard of allocated tutees
- view assigned student list
- schedule physical and virtual meetings
- record meeting outcomes
- upload meeting recordings
- view student documents
- add document comments
- message assigned students
- create blog posts and comment on blogs
- manage own profile

### Staff Features

- create user accounts
- view student and tutor data
- allocate tutors to students
- reallocate tutors
- bulk allocation
- view student and tutor dashboards in read-only mode
- manage own profile

### Admin-Level Staff Features

- all standard staff features
- reset user passwords
- view blog posts across the system
- view activity logs
- view statistical reports
- view exception reports

### Reporting and Monitoring

- activity logging for important actions
- dashboard and report data for page views, browser usage, active users, and system activity
- exception reports for:
  - students without tutors
  - students with no interaction over 7 days
  - students with no interaction over 28 days
- scheduled inactivity warning emails to students and tutors

## Roles and Permissions

The system uses a role-based access model backed by `roles`, `permissions`, and `role_permissions`.

### Student

- access only student-facing pages and own data
- can message assigned tutor only
- can upload documents and view feedback

### Tutor

- access tutor-facing pages and allocated student data
- can manage meetings and document comments
- can message assigned students only

### Staff

- can manage users and tutor allocation workflows
- can view student and tutor dashboards
- cannot access admin-only reporting or password-reset features unless `is_admin = 1`

### Admin-Level Staff

- standard staff permissions
- plus reporting, system monitoring, and password reset access

## How It Works

### Frontend

- `index.php` redirects users into the frontend login page
- frontend config is computed dynamically from the current project path in `Frontend/assets/js/core/config.js`
- page-specific logic lives in `Frontend/assets/js/features/`
- shared frontend runtime logic lives in `Frontend/assets/js/core/`

### Backend

- all API traffic goes through `Backend/api/index.php`
- requests are routed by `controller` query parameter
- protected routes pass through authentication middleware before dispatch
- controllers handle CRUD and custom actions
- services handle validation, notifications, inactivity checks, document preview, and permissions

### Database

- the schema is provided in the root SQL file:

```text
etutor.sql
```

- the backend uses environment-configured database credentials from `Backend/.env`

## Getting Started

### 1. Place the project in XAMPP htdocs

Example:

```text
C:\xampp\htdocs\eTutor
```

### 2. Import the database

Create a database, then import:

```text
etutor.sql
```

### 3. Install backend dependencies

Run inside `Backend/`:

```bash
composer install
```

### 4. Configure environment variables

Edit:

```text
Backend/.env
```

### 5. Start Apache and MySQL

Use XAMPP Control Panel, then open:

```text
http://localhost/eTutor/
```

## Environment Variables

The backend reads configuration from `Backend/.env`.

### Database

- `ETUTOR_DB_HOST`
- `ETUTOR_DB_PORT`
- `ETUTOR_DB_NAME`
- `ETUTOR_DB_USER`
- `ETUTOR_DB_PASS`

### JWT

- `ETUTOR_JWT_ACCESS_SECRET`
- `ETUTOR_JWT_REFRESH_SECRET`

### Mail

- `ETUTOR_MAIL_ENABLED`
- `ETUTOR_MAIL_TRANSPORT`
- `ETUTOR_MAIL_FROM`
- `ETUTOR_MAIL_FROM_NAME`
- `ETUTOR_SMTP_HOST`
- `ETUTOR_SMTP_PORT`
- `ETUTOR_SMTP_SECURE`
- `ETUTOR_SMTP_USERNAME`
- `ETUTOR_SMTP_PASSWORD`
- `ETUTOR_MAIL_EXPOSE_RESET_TOKEN`
- `ETUTOR_MAIL_ASYNC`
- `ETUTOR_ALLOCATION_MAIL_ENABLED`

### Other Runtime Flags

- `ETUTOR_PHP_CLI`
- `ETUTOR_REQUIRE_ACTIVE_TUTOR_ON_STUDENT_LOGIN`

Example safe template:

```env

ETUTOR_DB_USER=root
ETUTOR_DB_PASS=

ETUTOR_JWT_REFRESH_SECRET=change_this_refresh_secret

ETUTOR_MAIL_ENABLED=true
ETUTOR_MAIL_TRANSPORT=smtp

ETUTOR_MAIL_FROM_NAME=eTutor

ETUTOR_SMTP_PORT=587
ETUTOR_SMTP_SECURE=tls
ETUTOR_SMTP_USERNAME=your_username

ETUTOR_REQUIRE_ACTIVE_TUTOR_ON_STUDENT_LOGIN=false
```

## Background Tasks

The project includes backend task scripts for notifications and inactivity monitoring.

### Inactivity Warning Job

Used to send warning emails to students and tutors after inactivity thresholds.

Manual run:

```bash
php Backend/tasks/run_inactivity_warning.php 28
php Backend/tasks/run_inactivity_warning.php 7
```

Windows batch helper:

```text
Backend/tasks/run_inactivity_warning.bat
```

### Async Notification Workers

- `Backend/tasks/send_allocation_notification.php`
- `Backend/tasks/send_notification.php`

These are used by backend services to offload email sending from user-facing request time where configured.

### Smoke Test Support

- `Backend/tasks/smoke_matrix.php`

Useful for internal verification and evidence collection.
